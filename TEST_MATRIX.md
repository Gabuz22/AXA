# Test Matrix — Gabriel AXA v2.3.0

*Document unique de la phase de test conseillers. Publié avec la version de test ; la publication
du message dans le groupe Matrix reste une action humaine.*

> Outil **indépendant et non officiel**, non affilié ni validé par AXA. Construit à partir de
> documents accessibles publiquement. Les notices contractuelles et les sources officielles font
> foi. Toute information doit être vérifiée humainement avant une réponse ou une recommandation
> au client.

---

## 1. Message à publier dans Matrix (copier tel quel)

> **Gabriel AXA — phase de test conseillers 🧪**
>
> J'ai construit un outil de recherche dans la base contractuelle AXA (9 contrats, 493 faits
> sourcés) : tu poses ta question comme à un collègue, il te renvoie la garantie, l'exclusion ou
> la condition — **avec la notice et la page qui font foi**. Rien n'est inventé : si l'info n'est
> pas dans la base, il le dit.
>
> 👉 **https://gabuz22.github.io/AXA/**
>
> Teste-le avec tes **vraies questions de rendez-vous**, avec ou sans IA (parcours « Utiliser avec
> une IA » : un lien à coller dans ChatGPT/Claude/Gemini). Tes retours construisent la prochaine
> version : onglet **« Tester Gabriel AXA »** dans le site, un bouton copie tout — colle-le ici.
>
> ⚠️ Outil personnel, indépendant et non officiel (non validé par AXA). Pendant le test : vérifie
> toujours la notice avant d'utiliser une réponse avec un client, et **ne saisis aucune donnée
> client nominative** (ni dans le site, ni dans une IA).

## 2. Mode d'emploi (moins de deux minutes)

1. Ouvre **gabuz22.github.io/AXA** — la barre de recherche est au centre de l'accueil.
2. Tape ta question **comme tu la poserais à un collègue** (« délai de carence Avizen »,
   « le décès accidentel est-il couvert par MasterLife ? »).
3. Chaque réponse affiche sa **source** : clique « 📄 Notice p.X » pour ouvrir le PDF à la page.
4. Badge « **meilleur résultat** » = le plus proche de tes termes ; badge « **correspondance
   faible** » = la question est probablement hors du périmètre des 9 contrats, ne t'appuie pas
   dessus tel quel.
5. Pour travailler avec ton IA : onglet **« Utiliser avec une IA »**, copie le mini-prompt,
   colle-le dans ChatGPT / Claude / Gemini, puis pose tes questions normalement.

## 3. Cinq recherches pour commencer

1. `délai de carence` — condition transversale, plusieurs contrats.
2. `exclusions décès` — exclusions typées, sourcées.
3. `rachat possible ?` — question naturelle, contrats d'épargne.
4. `invalidité IPT` — définition + garanties (synonymes gérés).
5. Compare **Avizen** et **Avizen Pro** dans le **Comparateur** — ce que chacun apporte en propre.

## 4. Utilisation avec une IA — consignes

- Le parcours est **indépendant du fournisseur** : ChatGPT, Claude, Gemini, Copilot, Mistral…
- L'IA lit d'abord `https://gabuz22.github.io/AXA/ia/start.html` (protocole : citer contrat +
  notice + page, signaler les absences, jamais de mémoire générale sur AXA).
- **Aucune donnée client nominative** dans les questions (nom, coordonnées, n° de contrat client).
- Une réponse d'IA se vérifie **toujours** sur la notice avant usage client.

## 5. Protocole d'une semaine

| Jour | Action |
|---|---|
| J0 | Publication du message dans Matrix. Testeurs visés : 3 à 5 conseillers volontaires. |
| J0–J6 | Usage réel : chacun reproduit ses vraies questions de RDV (sans remplacer sa pratique). |
| J3 | Relance à mi-parcours : « qu'est-ce qui a manqué ? » — les retours « info manquante » d'abord. |
| J7 | Clôture : chaque testeur copie ses retours (`#/tester` → « Copier tous mes retours ») et les colle dans Matrix. |

**Centralisation des retours** : le fil Matrix est l'unique canal. Les retours sont saisis dans
l'app (onglet « Tester Gabriel AXA » — stockés localement dans le navigateur du testeur, rien
n'est envoyé automatiquement), puis copiés-collés volontairement dans le fil. Aucune donnée
client, aucun suivi intrusif, aucune plateforme analytique.

## 6. Ce que le protocole doit permettre d'observer

- combien de conseillers volontaires / combien ont réellement ouvert le site ;
- combien de retours structurés reçus (formulaire `#/tester`) ;
- informations trouvées ou non (champ « as-tu trouvé »), temps perçu vs pratique habituelle ;
- fonctionnalités réellement utilisées (recherche seule ? comparateur ? IA ?) ;
- réutilisation spontanée après le premier essai ;
- usage avec ou sans IA, et fiabilité perçue des réponses IA ;
- erreurs, incompréhensions, éléments manquants signalés ;
- intention déclarée de continuer à s'en servir.

## 7. Critères de décision en fin de semaine

**Test concluant si :**
- ≥ 3 testeurs ont fait au moins une vraie recherche ;
- ≥ 1 information réellement utilisée en préparation ou en RDV (vérifiée sur la notice) ;
- la majorité des retours dit « plus rapide » ou « pareil » (pas « moins pratique ») ;
- ≥ 1 testeur exprime l'intention de continuer.

**Signaux d'alerte (à traiter avant d'élargir) :**
- réponses perçues comme fausses ou sources ne correspondant pas ;
- confusion sur le statut de l'outil (cru officiel AXA) malgré les mentions ;
- une IA qui invente malgré le protocole START (noter le modèle et la question).

**Élargissement au-delà du groupe restreint** : uniquement après revue des retours ET validation
des points juridiques/conformité (droits de rediffusion des notices, statut de l'outil) — la
propreté technique ne vaut pas autorisation de diffusion large.

## 8. Limites connues de la version de test

- 9 contrats couverts ; tout le reste est **hors périmètre** (l'outil le signale désormais).
- Chiffres réglementaires (barèmes, abattements) : **pointeurs** vers les sources officielles,
  jamais stockés.
- Les analyses « Inspecteur » sont étiquetées **« analyse IA · à valider »** — jamais des faits.
- Fautes de frappe : les accents sont neutralisés ; une vraie faute (« carrence ») donne un
  résultat marqué « correspondance faible », pas de correction automatique.
